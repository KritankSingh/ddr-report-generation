1. WHAT WAS BUILT
================================================================================

A complete AI-powered pipeline that:
  • Reads two property inspection PDFs (Inspection Report + Thermal Images)
  • Extracts text and images from both documents
  • Sends data to Claude AI for intelligent analysis
  • Generates a professional 7-section Detailed Diagnostic Report (DOCX)
  • Outputs formatted document with embedded images and recommendations

EXECUTION FLOW:
  Input PDFs → Extraction → LLM Analysis → DOCX Generation → Professional Report

================================================================================
2. SYSTEM ARCHITECTURE
================================================================================

4 CORE PYTHON MODULES:

  main.py (2.3 KB)
    └─ Entry point
    └─ Orchestrates the entire pipeline
    └─ Usage: python main.py "input1.pdf" "input2.pdf" "output.docx"

  extractor.py (1.8 KB)
    └─ PDF parsing and extraction
    └─ Uses PyMuPDF (fitz) for text and images
    └─ Returns: {text: str, images: list[bytes]}

  llm_processor.py (5.0 KB)
    └─ Claude API integration
    └─ Structures findings into 7 DDR sections
    └─ JSON parsing with markdown code block stripping
    └─ Returns: structured DDR dict + images

  report_generator.py (4.6 KB)
    └─ DOCX document generation
    └─ Uses python-docx (follows .claude/Skills/docx.md pattern)
    └─ Embeds images, formats sections, outputs .docx file

5. 7-SECTION DDR OUTPUT
================================================================================

Generated DOCX contains:

  1. PROPERTY ISSUE SUMMARY
     └─ High-level overview of all findings (2-3 sentences)

  2. AREA-WISE OBSERVATIONS
     └─ Per-location breakdown with severity and thermal references
     └─ Images embedded under relevant observations

  3. PROBABLE ROOT CAUSES
     └─ Analysis of inferred causes based on evidence

  4. SEVERITY ASSESSMENT
     └─ Overall rating (Critical/High/Medium/Low) with reasoning

  5. RECOMMENDED ACTIONS
     └─ Ordered list of specific remediation steps

  6. ADDITIONAL NOTES
     └─ Context, methodology, and relevant information

  7. MISSING OR UNCLEAR INFORMATION
     └─ Explicit data gaps listed (never left blank)

7. PROBLEMS FIXED
================================================================================

PROBLEM 1: Raw JSON in Document Output
  Issue:   LLM wrapped response in markdown code blocks (```json...```)
  Fix:     Added markdown stripping logic in llm_processor.py
  Result:  Clean JSON parsing, no markdown visible in output

PROBLEM 2: Sections 2, 3, 5 Showing "Not Available"
  Issue:   Token limit (2048) too small for large DDRs
  Fix:     Increased max_tokens to 4096 in Claude API call
  Result:  All sections fully populated with real content

PROBLEM 3: Unicode/Encoding Errors on Windows
  Issue:   Special characters failing in Windows console
  Fix:     Set UTF-8 encoding in main.py
  Result:  Smooth cross-platform execution

10. KEY DECISIONS & DESIGN PATTERNS
================================================================================

1. MODULAR ARCHITECTURE
   Why: Easy to test, modify, and extend individual components
   How: Separate extraction, processing, and generation into distinct modules

2. CLAUDE HAIKU (not Opus/Sonnet)
   Why: Fast response, low cost, sufficient for structured analysis
   How: Trade-off between capability and efficiency for text-to-JSON task

3. JSON SCHEMA ENFORCEMENT
   Why: Ensures predictable output structure across all reports
   How: System prompt defines strict JSON schema; parsing validates

4. MARKDOWN CODE BLOCK STRIPPING
   Why: Handle LLM tendency to wrap code in ```json...```
   How: Regex extraction before JSON parsing

5. ZERO-HALLUCINATION DESIGN
   Why: Every claim must be traceable to source documents
   How: System prompt explicitly forbids inventing facts

6. GRACEFUL MISSING DATA HANDLING
   Why: Incomplete data is normal; better to mark explicitly than omit
   How: Use "Not Available" and list gaps in Section 7
